# 产物收尾决策（Artifact Finishing）

## 概述

**"差不多好了"不是状态，是逃避。**

本 skill 防止智能体在产物完成后停留在模糊状态——既不锁定也不废弃，导致下游无法判断能否使用、版本无法追溯、变更无法管理的失败模式。

**核心原则：** 产物完成后必须做出明确决策，四选一。不选择 = 未完成。

## 铁律

```
NO AMBIGUOUS COMPLETION STATE
```

如果产物的状态不是归档、发布、废弃、回滚之一，它就不是完成。停下。做出决策。

## 执行规则

1. **前置条件检查**
   - 产物必须先经过 `review-before-acceptance` 审查且全部通过
   - 未通过审查的产物不能进入收尾决策

2. **四选一决策**

   | 状态 | 定义 | 触发条件 | 后续约束 |
   |------|------|---------|---------|
   | **归档** | 锁定版本，不再修改 | 产物完整、审查通过、无下游依赖 | 版本冻结，后续变更必须创建新版本 |
   | **发布** | 进入下游使用阶段 | 产物完整、审查通过、有下游依赖 | 变更必须走变更传播协议 |
   | **废弃** | 明确标记不再使用 | 产物被替代、需求取消、或方向错误 | 标注 `[已废弃]` 及原因，保留只读 |
   | **回滚** | 回退到上一稳定版本 | 当前产物有严重缺陷、无法修复 | 废弃当前版本，激活上一稳定版本 |

3. **决策必须声明**
   - 在产物文件头部标注最终状态
   - 说明决策依据（审查结论、用户确认、缺陷评估）
   - 如果是发布状态，标注下游依赖清单

4. **不选择 = 未完成**
   - 如果用户或团队无法做出决策 → 产物视为未完成
   - 未完成的产物不能进入下游、不能对外发布、不能作为上游输入

## 各状态的执行动作

### 归档

```
审查通过
    ↓
在产物头部添加：「状态：已归档 | 归档日期：YYYY-MM-DD」
    ↓
锁定版本号，不再接受补丁
    ↓
如需变更 → 创建新版本（v{旧+1}.0），旧版本保持只读
```

### 发布

```
审查通过
    ↓
在产物头部添加：「状态：已发布 | 发布日期：YYYY-MM-DD | 下游依赖：[清单]」
    ↓
激活变更传播协议：任何修改必须评估下游影响
    ↓
下游产物引用此版本时，必须在头部声明引用范围
```

### 废弃

```
决策确认
    ↓
在产物头部添加：「状态：[已废弃] | 废弃日期：YYYY-MM-DD | 原因：[说明]」
    ↓
如有替代产物，在废弃标注中提供替代链接
    ↓
文件保持只读，不删除（保留历史可追溯性）
    ↓
通知所有下游引用方更新引用
```

### 回滚

```
严重缺陷确认（无法在当前版本内修复）
    ↓
评估上一稳定版本的完整性
    ↓
在产物头部添加：「状态：[已回滚] | 回滚日期：YYYY-MM-DD | 原因：[缺陷描述] | 回退至：vX.Y」
    ↓
废弃当前版本，激活上一稳定版本
    ↓
通知所有下游引用方版本变更
    ↓
如上一稳定版本也受影响 → 启动级联回滚评估
```

## 危险信号 —— 停下并重新评估

- 产物审查通过了，但没说最终状态是什么
- 用户说"先放着吧"——这是未决策，必须追问四选一
- 产物标了"完成"但没有状态标注
- 废弃的产物被下游继续引用（废弃通知未到位）
- 回滚后未通知下游，导致下游基于错误版本工作
- 归档的产物被偷偷修改（未创建新版本）

**以上任何一条都意味着：收尾决策失效，产物状态失控。停下。明确状态。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "后面可能还要改" | 需要后续变更 = 发布状态，不是"不决策"。 |
| "用户没说要归档还是发布" | 决策是你的责任。提供选项，用户选一个。 |
| "先标记完成，状态后面再定" | 没有状态的"完成"是谎言。 |
| "废弃太浪费了" | 保留错误产物比废弃它更浪费下游的时间。 |
| "回滚会影响团队士气" | 基于缺陷版本推进会浪费更多资源。 |
| "这个产物很小，不需要这么正式" | 小产物也有状态。归档它。 |

## 快速参考

| 情况 | 行动 |
|------|------|
| 产物完整、无下游依赖 | 归档 |
| 产物完整、有下游等待 | 发布，标注下游清单，激活变更传播 |
| 产物被新版本替代 | 废弃旧版本，标注替代链接，通知下游 |
| 产物方向错误 | 废弃，标注原因，不保留"备用"副本 |
| 当前版本有不可修复缺陷 | 回滚到上一稳定版本，通知所有下游 |
| 用户说"先放着" | 追问："是需要发布、归档、还是暂时废弃？" |
| 无法决定状态 | 视为未完成，返回审查或补充信息 |

## 必需子技能

在进入收尾决策之前：
- **必需子技能：** 使用 `review-before-acceptance` —— 未审查的产物不能做收尾决策

在决策为"发布"状态时：
- **必需子技能：** 使用 `artifact-workflow` —— 激活变更传播协议，管理下游依赖

在决策为"废弃"或"回滚"状态时：
- **必需子技能：** 使用 `clean-evolution` —— 干净地处理旧版本，不堆积历史包袱

> 注：本 skill 的「四选一」是强制决策框架。不存在第五种状态，"待定"不是合法状态。
