# 先计划后执行（Plan Before Execution）

## 概述

没有计划的执行是在用资源赌博。

**核心原则：** 没有书面计划就不能开工。计划是意图与执行之间的契约。

## 铁律

```
没有书面计划，不得执行
```

如果没有计划，你就不能开始执行。停下。先写计划。

## 执行前强制检查清单（必须逐条回答）

在每次执行任何操作之前，你必须在内心完成以下检查，并在行动中体现结果：

- [ ] **计划存在性**：我是否已检查是否存在书面计划（PLAN.md、步骤清单、规格文档或任何步骤分解）？
- [ ] **无计划处理**：如果没有书面计划，我是否已停止执行并加载 `writing-plans` 来创建计划？
- [ ] **用户确认**：如果用户说"跳过计划，直接做"，我是否已再次确认："没有书面计划，我应该直接开始吗？"
- [ ] **计划审阅**：如果计划存在，我是否已加载 `executing-plans` 并按计划逐条执行？
- [ ] **大任务拆分**：如果计划包含超过 10 个检查项/步骤/文件，我是否已将任务拆分而不是自行减项？

**以上任何一条未满足，都不得继续执行。**

## 执行规则

1. **先检查计划**：在任何执行之前，检查是否存在书面计划（PLAN.md、待办清单、规格文档，或任何步骤分解文档）。
2. **如果计划存在**：加载并审阅它。按照 executing-plans 逐条执行。
3. **如果计划不存在**：立即停止执行。加载 writing-plans 来创建计划。
4. **计划质量检查**：计划必须包含具体步骤，不能只是模糊的意图。「做 X」不是计划。「第一步：…… 第二步：……」才是计划。
5. **用户覆盖**：只有当用户明确说「跳过计划，直接做」时，你才能在没有计划的情况下继续。

## 危险信号 —— 停下

- 在检查是否有计划之前就开始实施
- "我先做着看"
- 用户要求执行你就立刻开始
- 把模糊的想法当成计划
- 因为「看起来简单」就继续

**以上任何一条都意味着：你即将在没有契约的情况下执行。停下。先制定计划。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "这是个简单任务" | 简单任务有简单的计划。写下来。 |
| "用户要得急" | 没有计划的匆忙比计划更花时间。 |
| "我边做边计划" | 执行中做计划是原地打转。 |
| "我们以前做过" | 过去的经验 ≠ 当前的计划。记录下来。 |
| "用户没要求计划" | 制定计划是你的责任，不是用户的请求。 |

## 快速参考

| 情况 | 行动 |
|------|------|
| 计划文件存在（PLAN.md 等） | 审阅 → 使用 executing-plans |
| 没有计划文件 | 停下 → 使用 writing-plans |
| 用户说「直接做」 | 再问一次："没有书面计划，我应该直接开始吗？" |
| 计划模糊 / 有占位符 | 拒绝该计划 → 请用户澄清或重写 |

## 必需子技能

如果存在书面计划：
- **必需子技能：** 使用 `superpowers:executing-plans`

如果不存在书面计划：
- **必需子技能：** 使用 `superpowers:writing-plans`
