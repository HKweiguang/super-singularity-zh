# 验收标准先行（Acceptance Criteria First）

## 概述

**没有完成标准的产物等于没有完成。**

本 skill 防止智能体在未定义"怎么算完成"的情况下就开始生成内容，导致产物边界模糊、质量不可验证、用户反复要求修改的失败模式。

**核心原则：** 任何产物生成前，必须先有可验证的验收标准。内容是对标准的实现，不是对意图的猜测。

## 铁律

```
NO CONTENT WITHOUT ACCEPTANCE CRITERIA
```

如果验收标准不存在或不可验证，你就不能开始生成内容。停下。先写标准。

## 执行规则

1. **识别产物类型**：明确当前要生成什么（PRD / 设计稿 / 技术方案 / 测试计划 / 合规报告等）。
2. **提取验收标准**：在动笔之前，先列出该产物的验收标准。
   - 如果用户提供了标准 → 审阅并确认可验证性
   - 如果用户未提供标准 → 主动定义并征求用户确认
   - 如果上游文档定义了标准 → 提取并转化为当前产物的验收标准
3. **验证标准可执行**：每条标准必须满足：
   - **可观察**：结果可以被看到、测量或检查
   - **可判定**：给定相同输入，任何审查者都能得出相同结论
   - **原子性**：一条标准只验证一个维度，不堆叠多个断言
4. **绑定到产物头部**：将验收标准写入产物文件的 §8（或等效章节），作为审计的输入。
5. **按标准生成**：内容是对验收标准的实现。每完成一条标准，自检一次。

## 验收标准格式

使用 **Given-When-Then** 格式，确保每条标准都是可验证的行为描述：

```
- 给定 [前置条件 / 初始状态]
- 当 [触发动作 / 输入事件]
- 那么 [可观察的、可判定的结果]
```

### 示例（PRD）：
- 给定用户在登录页
- 当输入正确手机号并点击获取验证码
- 那么 60 秒内收到短信验证码

### 示例（设计稿）：
- 给定在 375px 宽度的移动设备上
- 当打开首页
- 那么首屏加载时间 < 2 秒，且布局无错位

### 示例（技术方案）：
- 给定系统并发用户达到 10,000
- 当用户同时提交订单
- 那么 99.9% 的请求在 200ms 内完成，且无数据不一致

### 示例（测试计划）：
- 给定已部署 v2.3 版本到 staging 环境
- 当执行完整回归测试套件
- 那么所有 P0 用例通过，P1 用例通过率 ≥ 95%

## 产物类型的通用验收维度

| 产物类型 | 必含验收维度 |
|---------|------------|
| PRD | 功能覆盖度、业务规则完整性、错误处理场景、性能基线 |
| 交互设计 | 用户流程闭环、状态转换完整性、异常路径覆盖 |
| UI 设计 | 响应式断点、可访问性标准、设计系统一致性 |
| 技术方案 | 接口契约、性能指标、安全基线、回滚策略 |
| 测试计划 | 覆盖率目标、环境要求、通过/失败判定标准 |
| 部署文档 | 前置条件清单、验证步骤、回滚指令 |

## 危险信号 —— 停下并重新评估

- 用户说"写个文档"，你直接开始写，没有先问"怎么算完成"
- 产物已经写了 50%，才发现缺少验收标准 → 已写内容可能全错
- 验收标准写成"功能完善"、"用户体验好"——不可判定
- 一条标准包含多个断言（"那么 A 且 B 且 C"）→ 失败时无法定位
- 标准只写了正向路径，没写边界条件和错误场景
- 生成内容时忽略已定义的验收标准

**以上任何一条都意味着：你在没有罗盘的情况下航行。停下。先定义标准。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "用户没要求验收标准" | 定义标准是你的责任，不是用户的请求。 |
| "我先写着看" | 没有标准的内容是猜测，猜测的返工成本更高。 |
| "这个产物不需要标准" | 任何可交付物都有"完成"的判定条件。找不到说明理解不够深。 |
| "标准可以后面补" | 事后补的标准会被已有内容反向合理化，失去客观性。 |
| "标准太细会限制创造力" | 标准定义边界，不限制边界内的实现方式。 |
| "简单产物不需要这么复杂" | 简单产物有简单的标准。写下来。 |

## 快速参考

| 情况 | 行动 |
|------|------|
| 用户提供了详细需求 | 将需求转化为 Given-When-Then 格式，用户确认后再动笔 |
| 用户需求模糊 | 用验收标准作为澄清工具——"如果我理解为 X，那么标准是 Y，对吗？" |
| 上游文档已有验收标准 | 提取并转化为当前产物的验收标准，检查一致性 |
| 产物涉及多个利益相关方 | 每个相关方至少有一条对应的验收标准 |
| 标准似乎无法验证 | 拆解为更小的标准，直到每条都可观察、可判定 |
| 用户说"直接做" | 再确认一次："没有验收标准，产出后如何判定是否满足需求？" |

## 必需子技能

在验收标准定义完成后、内容生成之前：
- **必需子技能：** 使用 `plan-before-execution` —— 确保有执行计划，标准与计划对齐

在内容生成完成之后：
- **必需子技能：** 使用 `evidence-before-claims` —— 逐条验证产物是否满足验收标准
- **必需子技能：** 使用 `artifact-review` —— 产物审查时以验收标准为审查清单

> 注：本 skill 的「验收标准」是节点工作流中的下游约束。上游文档中的需求定义是验收标准的输入，但不是替代。
