# 产物工作流（Artifact Workflow）

> 可自定义步骤的产物链管理 skill —— 提取「从想法到落地」的核心思想，让用户定义自己的工作流。

## 定位

本 skill 是 **一套协议 + 一份指南**。

| 层次 | 内容 | 说明 |
|------|------|------|
| **CORE（核心）** | 原则、协议、审计框架 | 与领域无关，任何工作流都必须遵守 |
| **CUSTOM（自定义）** | 工作流定义方法、模板编写指南 | 用户按指南定义自己的步骤 |

**使用前提**：`.artifacts/workflow.md` 是可选增强配置。存在时按自定义工作流执行；不存在时进入「通用执行模式」，核心原则仍然生效。

### 能力边界

- **编排 vs 执行**：artifact-workflow 只负责定义步骤顺序、管理依赖、追踪变更传播；具体生成执行由 AI 或 MCP 工具完成
- **超出 AI 能力的产物**（如二进制文件）：在 `workflow.md` 中声明 `工具依赖`，执行时触发对应 MCP 工具

## 1. 何时使用本 Skill

| 场景 | 处理方式 |
|------|---------|
| 用户要求生成/修改某类产物（如需求文档、设计稿、测试计划、合规报告等） | 静默检测 `.artifacts/workflow.md`：存在则按自定义步骤执行；不存在则进入通用执行模式，任务完成后询问是否建立正式工作流 |
| 用户想定义/调整自己的工作流 | 进入「工作流配置模式」，按第6节指南定义工作流 |
| 用户修改了某份产物，担心下游不一致 | 触发变更传播协议，扫描下游影响 |
| 用户想审计产物质量 | 按审计框架执行机械检查 + 语义检查 |
| 用户不清楚如何开始 | 直接询问用户业务需求，按通用执行模式推进；任务完成后提示可建立正式工作流 |

## 2. 执行任何步骤前的统一动作

无论用户要做什么，先执行：

```
1. 读取 AGENTS.md（如有）了解项目背景
2. 静默查找 .artifacts/workflow.md
   ├─ 存在 → 读取 workflow.md，确定当前步骤类型、上下游关系
   │         加载对应步骤模板（.artifacts/steps/{type}-step.md）
   │         按自定义工作流执行
   └─ 不存在 → 尝试 DAG 自动推导（如产物目录已存在）
                 ├─ 推导成功 → 按推导出的依赖关系执行
                 └─ 推导失败 → 进入「通用执行模式」（见 §2.1），暂不提及工作流配置
3. 按核心原则执行
```

**禁止行为**：workflow.md 不存在时，不得反问用户"是否创建工作流"来阻塞业务需求。

## 2.1 通用执行模式（无 workflow.md 时的默认行为）

当 `.artifacts/workflow.md` 不存在时，按以下通用原则推进：

```
1. 识别用户意图：明确产物类型（PRD / 设计稿 / 技术方案 / 测试计划等）
2. 按产物类型通用模板执行（见 §4 产物协议）
3. 产物头部至少包含：版本号、制定日期、上游文档引用（如有）
4. 执行审计框架的通用检查（结构完整性、可验证性）
5. 任务完成后 → 询问用户："是否建立正式工作流以提升后续一致性？"
```

**核心原则仍然生效**：
- 依赖可追溯：产物头部声明上游引用（即使只有用户输入）
- 变更必传播：后续如有上下游产物，仍执行传播协议
- 审计双轨制：格式检查由 AI 执行，脚本可用时叠加
- 契约驱动规范：编号、术语按项目已有约定或通用规范

## 必需子技能

本 skill 只负责产物链编排，所有执行纪律由以下 skill 强制执行：

- **在执行任何步骤之前：**
  - **必需子技能：** 使用 `plan-before-execution` —— 开工前检查是否有书面计划
- **在步骤执行期间：**
  - **必需子技能：** 使用 `complete-task-execution` —— 不跳过任何子任务地执行
- **在审计产物时：**
  - **必需子技能：** 使用 `audit-framework` —— 在 6 个维度和 3 个层级上进行系统性质量检查
- **在声称完成之前：**
  - **必需子技能：** 使用 `evidence-before-claims` —— 在验收前用证据验证
- **在发现上游定义中的缺陷时：**
  - **必需子技能：** 使用 `systematic-problem-solving` —— 在修补之前找到根因
- **在传播变更或演进设计时：**
  - **必需子技能：** 使用 `clean-evolution` —— 干净地替换旧的，绝不无决策地层叠
- **当工作流步骤涉及编写或修改代码时：**
  - **必需子技能：** 使用 `superpowers:subagent-driven-development` 或 `superpowers:test-driven-development` —— 代码实现属于 superpowers 的领域

## 3. 核心原则（不可违背）

### 3.1 四个不变原则

| 原则 | 含义 | 反模式 |
|------|------|--------|
| **依赖可追溯** | 每份产物必须在头部声明「引用了哪些上游文档、引用了哪些内容」 | 产物孤立存在，无法判断它基于什么输入 |
| **变更必传播** | 修改上游后，必须扫描下游影响、询问用户是否同步、拒绝则标注 `[待同步]` | 改了上游忘了改下游，导致实现漂移 |
| **审计双轨制** | 机械检查（格式、连续性、存在性）归脚本；语义检查（正确性、合理性）归 AI | 脚本做语义判断，或 AI 做穷举存在检查 |
| **契约驱动规范** | 跨产物共享的规则（编号、术语、状态）在「全局契约」中统一定义 | 各产物各自定义，导致不一致 |

### 3.2 三个工作模式

| 模式 | 触发条件 | 核心动作 |
|------|---------|---------|
| **新建模式** | 当前步骤类型的产物不存在 | 加载全局契约 → 解析上游 → 按模板生成 → 审计 → 写入 |
| **修改模式** | 当前步骤类型的产物已存在 | 加载现有产物 → 加载上游 → 扫描下游影响 → 执行修改 → 审计 → 更新头部 |
| **回改模式** | 执行中发现上游定义有缺陷/遗漏/矛盾 | 暂停 → 定位上游 → 评估下游影响 → 用户确认 → 级联修改 → 全量重新审计 |

## 4. 产物协议

### 4.1 产物头部标准

每份产物文件头部必须包含：

```markdown
# {产物名称}

**版本号**：v{主版本}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
| 文档 | 类型 | 引用范围 |
|------|------|---------|
| {上游文档路径/名称} | 规范继承 / 需求输入 / 设计输入 | {具体章节/编号/范围} |
```

- `类型` 必须明确是**规范继承**还是**输入依赖**
- `引用范围` 必须具体到章节、编号或功能点，不能写"全文"
- 修改时如上游有更新，必须同步更新此表

### 4.2 产物边界规则

| 规则 | 说明 |
|------|------|
| 不写已在上游定义的内容 | 如需求产物不写技术实现 |
| 不写属于下游职责的内容 | 如需求产物不写测试用例 |
| 发现越界内容时标记并提示迁移 | 不在当前产物中默默兼容 |

### 4.3 审计协议（双轨制 + 轮次控制）

#### 审计阶段

```
机械检查（脚本）→ 未通过 → 修复 → 必须重新机械检查
   ↓ 通过
语义检查（AI）→ 未通过 → 修复 → 重新语义检查
   ↓ 通过
审计完成
```

#### 硬性规则

1. **机械未通过禁止启动语义检查。**
2. **修复后必须复验。** 机械修复 → 重新阶段一；语义修复 → 重新阶段二。
3. **单份产物审计+修复最多进行 3 轮**（含初次审计）。第3轮仍不通过 → 停止自动修复，向用户呈现剩余问题，请求人工决策。
4. **删除优于标注。** 审计发现多余、错误、越界内容时，优先直接删除。例外：上游已修改但用户拒绝同步 → 标注 `[待同步]`。

#### 修复后复验规则

| 修复类型 | 复验要求 |
|---------|---------|
| 机械修复（格式、编号、结构） | 对该产物重新执行**完整机械检查**（阶段一），通过后方可进入语义复验 |
| 语义修复（内容、逻辑） | 仅对该产物重新执行**语义检查**（阶段二） |
| 混合修复（格式+内容） | 先完整机械检查，通过后语义检查 |

#### 脚本与 AI 分工

| 归属 | 职责 |
|------|------|
| **脚本**（audit-engine） | 格式、存在性、连续性、编号映射、契约追溯——结果只有通过/不通过 |
| **AI** | 语义判断（正确性、合理性、满足度）——结果可能有"存疑" |

**规则**：能确定性描述的检查 → 脚本；需要业务上下文理解的检查 → AI。禁止降级规则精度让脚本覆盖。

## 5. 变更传播协议

### 5.1 修改上游时的强制流程

```
修改上游文档
    ↓
扫描下游影响（首选 DAG 精确分析，fallback 手动分层扫描）
    ↓
列出影响范围清单（直接影响 → 间接影响 → 无影响），按优先级排序
    ↓
询问用户是否级联修改
    ├─ 用户确认 → 同步修改所有受影响下游
    └─ 用户拒绝 → 在受影响下游标注 [待同步]
    ↓
重新验证（全量审计，禁止缩减维度）
```

**DAG 精确分析**：
- 如产物目录已建立，优先运行 `dag-engine.py --impact {修改文件} --dir {产物目录}`
- 输出直接影响（波次 N+1）、间接影响（波次 > N+1）、无影响三类清单
- 精确分析不可替代人工判断，但可大幅减少模糊扫描范围

### 5.2 修改下游时的边界检查

```
修改下游产物
    ↓
读取所有上游文档
    ↓
逐条比对修改内容是否在上游定义范围内
    ├─ 未超出 → 直接修改
    └─ 超出 → 立即停止，先改上游
```

## 6. 如何自定义工作流

项目在工作目录下创建 `.artifacts/` 目录，按以下结构组织：

```
project-root/
├── AGENTS.md
└── .artifacts/
    ├── workflow.md              ← 工作流定义（必须）
    ├── steps/                   ← 步骤模板（每种步骤一个文件）
    │   ├── requirement-step.md
    │   ├── design-step.md
    │   └── implementation-step.md
    ├── top-level/               ← 全局契约（跨产物共享规则）
    │   └── project-top-level.md
    └── audit-rules.yaml         ← 可选：自定义审计规则
```

## 7. 工具调用

本 skill 约束 AI 行为，实际检查由以下脚本执行。

### 7.1 审计引擎（机械检查）

```bash
# 单文件审计
python3 scripts/audit-engine/cli.py --file <产物路径> --type <产物类型>

# 目录批量审计
python3 scripts/audit-engine/cli.py --dir <产物目录> --type <产物类型>

# 生成模型检查清单（脚本穷举检查点 + AI 逐条判断）
python3 scripts/audit-engine/cli.py --model-checklist --type <产物类型>
```

**产物类型**：从配置动态加载，当前支持 `prd/interaction/ui/tech/test/generic` 及用户自定义类型。

**配置目录**：
- 插件默认：`scripts/audit-engine/config/`
- 项目自定义：`.singularity/config/`
- CLI 覆盖：`--config-dir <路径>`

### 7.2 DAG 引擎（依赖分析）

```bash
# 可视化产物依赖图
python3 scripts/dag-engine.py --dir <产物目录> --visualize

# 检测循环依赖
python3 scripts/dag-engine.py --dir <产物目录> --detect-cycles

# 变更影响分析
python3 scripts/dag-engine.py --dir <产物目录> --impact <修改文件>

# 执行波次调度 + 关键路径
python3 scripts/dag-engine.py --dir <产物目录> --execution-plan
```

### 7.3 语义检查清单

按产物类型加载对应检查清单，AI 逐项判断：

```
scripts/audit-engine/checklists/
├── generic-checklist.md      ← 通用（所有产物类型回退）
├── prd-checklist.md          ← 产品需求
├── tech-checklist.md         ← 技术方案
└── test-checklist.md         ← 测试计划
```

用户自定义：`.singularity/config/checklists/{type}-checklist.md`

### 7.4 Skill-脚本协作流程

```
AI 按 artifact-workflow 生成/修改产物
    ↓
运行 audit-engine --file 进行机械检查
    ↓
机械通过 → AI 按 checklists/{type}-checklist.md 进行语义检查
    ↓
全部通过 → 进入 review-before-acceptance
    ↓
审查通过 → artifact-finishing 四选一决策
```
