# 干净演进（Clean Evolution）

## 概述

**按设计演进，而不是靠积累。新替换旧。旧不成为新的锚。**

本 skill 防止智能体把新内容或逻辑层叠在旧内容之上，创造出一个既服务不了旧设计也服务不了新设计的混乱混合体的失败模式。当需求变化时，智能体必须有意识地决定保留什么、删除什么、替换什么——而不是默默堆积补丁。

## 核心原则

**默认保留就是设计层面的腐化。**

当新旧在没有明确意图的情况下共存时，结果比单独的任何一个都更糟。每块保留的旧内容都必须在新设计下证明其存在的合理性。如果不能，就必须去掉。

## 执行规则

1. **评估关系**：在做任何更改之前，明确评估现有内容与新方法的关系。什么服务于新设计？什么与之冲突？
2. **只保留服务于新设计的**：只有当旧内容直接支持新设计时才保留。不要仅仅因为存在或"还能用"就保留。
3. **删除冲突的**：如果旧内容与新方法矛盾或重复，删除它。不要把过时的内容、废弃的逻辑或冲突的模式留在原地。
4. **不决策不层叠**：在旁边添加新内容是一个深思熟虑的选择，不是默认值。如果你创建了新实现，旧实现必须被处理——删除、弃用或显式桥接。
5. **声明你的决策**：演进之后，明确告诉用户保留了什么、删除了什么、替换了什么。不要让它隐含不明。

## 危险信号 —— 停下并重新评估

- 逐步修改旧内容来匹配新设计 → 创造一半旧一半新的混乱
- 用户没要求就「为了向后兼容而保留"
- 新内容与旧内容并存，做着类似或重叠的事情
- 添加新组件/章节而不处理旧的那个
- 不说明哪些部分被替换了、哪些保留了
- "我先留着旧的以防万一"

**以上任何一条都意味着：你在积累技术债。停下。干净地演进。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "旧内容还能用" | 能用 ≠ 在新设计下正确。冲突的现有内容仍然是缺陷。 |
| "用户没说删除" | 干净演进是你的责任。如果不确定，问——但不要默认囤积。 |
| "逐步迁移更安全" | 除非用户明确要求逐步迁移，否则默认 cleanly 替换。 |
| "删除有风险" | 新旧混合比 clean 替换更危险，因为行为变得不可预测。 |
| "我可能会破坏什么" | 那就是验证的用途。测试 clean 替换之后，不要通过留下垃圾来逃避。 |

## 当用户说「改这个」时

即使用户把它框定为一个小改动：

1. **不要**默认把它当作增量补丁。
2. **要**问：「这是替换现有方法，还是两者应该共存？如果两者共存，迁移计划是什么？」
3. **要**在实施之前提出 clean evolution 计划。

## 快速参考

| 情况 | 行动 |
|------|------|
| 新架构替换旧架构 | 完全删除旧实现 |
| 新功能与现有功能重叠 | 决定：合并、替换，还是以明确边界共存 |
| 重组现有工作 | 不要在新模式旁边放旧模式——迁移或保持一致 |
| 用户说「也支持 X" | 检查 X 是否与当前设计冲突。如果是，提出重新设计。 |
| 不确定是否保留旧的 | 问用户。不要默认保留。 |

## 必需子技能

在演进完成之后：
- **必需子技能：** 使用 `evidence-before-claims` —— 在声称完成之前验证新设计有效且旧冲突已消除
