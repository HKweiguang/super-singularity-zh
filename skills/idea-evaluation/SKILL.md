# 想法评估与方案融合分析

> 用户的第一句话是起点，不是规格说明书。未经 gap 分析，禁止基于模糊想法输出完整方案。

## 铁律

```
未经评估，禁止基于模糊想法输出产物或修改代码
```

## 触发条件

- 用户提出"我想做一个...""能不能加...""如果改成..."
- 用户询问"这个想法可行吗""和现有方案冲突吗"
- 用户给出模糊需求，需要先分析再细化
- 用户要求"帮我看看这个需求怎么落地"

## 执行步骤

### 1. 加载现有方案

```
1. 读取 AGENTS.md（如有）了解项目背景
2. 搜索项目内相关产物（通过文件名特征或上游文档表格识别）
3. 定位可能受影响的模块和上下游产物
4. 如无现有产物 → 明确告知用户当前为"从 0 开始"，建议先建立全局契约
```

### 2. Gap 分析

将用户的想法与现有方案比对，识别：

| 维度 | 分析内容 |
|------|---------|
| **信息缺口** | 边界条件、权限规则、异常场景、数据模型、状态流转等是否缺失 |
| **功能重叠** | 与现有功能是否重复、冲突或可复用 |
| **范围蔓延** | 是否超出当前模块/产品的职责边界 |
| **技术风险** | 是否需要引入新技术、新依赖，或涉及复杂架构调整 |

### 3. 影响范围评估

| 维度 | 分析内容 |
|------|---------|
| **直接影响** | 需要新建或修改哪些产物 |
| **全局影响** | 是否需要调整全局术语、状态值、编号规则、模块职责 |
| **下游影响** | 对后续产物的连锁变更 |
| **依赖影响** | 是否需要其他团队/模块配合 |

### 4. 输出评估结论

评估报告必须包含：

- **可行性判断**：可行 / 需调整后可行 / 当前不可行（说明原因）
- **融合方式**：
  - **独立新增**：作为新模块，不影响现有功能
  - **扩展现有**：在现有模块上增加功能点
  - **替换现有**：废弃旧功能，用新方案替代（需评估迁移成本）
- **缺失信息清单**：必须补全的关键信息，列成问题清单询问用户
- **建议后续步骤**：
  - 信息完备 → 进入产物生成流程（加载 artifact-workflow）
  - 信息缺失 → 先补充需求
  - 技术不确定 → 建议先输出预研产物

## 输出格式

```markdown
# 想法评估报告

## 1. 现有方案快照

- 已发现的产物：[列出相关产物]
- 关键约束：[全局契约中与本想法相关的规则]

## 2. Gap 分析

| 维度 | 发现 | 严重程度 |
|------|------|---------|
| 信息缺口 | ... | 高/中/低 |
| 功能重叠 | ... | 高/中/低 |
| 范围蔓延 | ... | 高/中/低 |
| 技术风险 | ... | 高/中/低 |

## 3. 影响范围

- 直接影响：[产物清单]
- 全局影响：[是否需要调整全局契约]
- 下游影响：[连锁变更清单]

## 4. 评估结论

- **可行性**：可行 / 需调整后可行 / 当前不可行
- **融合方式**：独立新增 / 扩展现有 / 替换现有
- **缺失信息**：[问题清单，向用户提问]
- **建议后续**：[下一步行动]
```

## 危险信号 —— 停下

- 用户说"我想做..."，你立刻输出完整方案
- 未经现有产物比对，直接判断"这个想法可行"
- 对模糊需求不做 gap 分析，直接开始设计
- 评估结论中没有「缺失信息清单」，强行推进
- 用户说"先做着看"，你不做评估就服从

**以上任何一条都意味着：你正在基于假设行动。停下。先评估。**

## 关键原则

1. **禁止直接输出完整方案**：未经 gap 分析确认，禁止基于模糊想法输出完整产物
2. **必须引用现有方案**：评估结论必须与现有产物中的具体章节/功能点/规则进行比对
3. **明确标注不确定性**：对于无法从现有产物中确认的部分，标注为「待确认」，不要假设
4. **对话式提问**：关键信息缺失时，发起 1-3 个问题，提供选项让用户选。用户说"跳过"→记录最小假设继续
