# 审计框架（Audit Framework）

## 概述

质量不是假设出来的。它是检查出来的。

**核心原则：** 每份产物在验收前必须通过系统性审计。

## 铁律

```
没有系统性审计，不得接受任何产物
```

## 检查维度

### 6 维度 × 3 层级

**6 维度：**
- **结构**：格式、组织、完整性
- **数据**：准确性、一致性、有效性
- **状态**：当前状态、生命周期阶段
- **错误**：处理、覆盖、边界情况
- **性能**：效率、可扩展性
- **安全**：访问、暴露、漏洞

**3 层级：**
- **L1 产物内部**：产物自身的结构和内容完整性
- **L2 相邻产物**：与直接上下游产物的一致性
- **L3 端到端链**：从头到尾的完整可追溯性

**2 跨切维度：** 版本、边界

### 检查项示例（文档/设计类产物）

以下检查项为**建议清单**，项目可根据自身需要开发机械检查脚本，或由 AI 人工执行。

| 维度 | 可机械化的检查项 | 需语义理解的检查项 |
|------|----------------|-----------------|
| **结构** | 标题层级连续、表格对齐、编号无跳号 | 章节完整性、内容边界清晰、必填项已填 |
| **数据** | 术语前缀一致、编号格式统一、链接可达 | 术语定义与全局契约一致、同一概念名称统一、引用上游存在 |
| **状态** | 版本号格式正确、日期有效 | 生命周期阶段合理、版本与里程碑对齐 |
| **错误** | 异常场景数量 ≥ 主场景数量 × 20% | 边界条件覆盖完整、错误处理有恢复路径 |
| **性能** | 响应时间基线有数值、并发指标有单位 | 基线合理可达成、可扩展性策略明确 |
| **安全** | 敏感字段已标注、权限矩阵无空行 | 数据分类合理、威胁模型覆盖主要攻击面 |

> **当前状态**：super-singularity-zh 暂未提供标准化机械检查脚本。各项目可按上表自行开发，或委托 AI 逐条人工执行。
>
> **裁剪原则**：产物类型不涉及某维度时，标注「不适用」并说明理由，不可直接跳过。

## 不一致时的根因判断

L2 和 L3 审计发现不一致时，**不能直接假设「当前产物错了」**。不一致有三种可能：

| 情况 | 根因 | 修复方向 |
|------|------|---------|
| 上游定义正确，当前产物理解错了 | 当前产物 | 修改当前产物 |
| 上游定义有问题，当前产物按错误上游执行 | 上游产物 | 触发回改模式，修改上游并传播 |
| 上下游都正确，但版本不匹配（上游已更新，当前产物未同步） | 版本同步 | 按变更传播协议更新当前产物 |

**判断流程：**

1. **先审上游**：在判断 L2/L3 不一致之前，确认上游产物是否已通过审计且状态稳定
   - 上游未审计 → 暂停当前审计，先完成上游审计
   - 上游审计未通过 → 上游有问题，当前产物的「不一致」可能是正确的反映

2. **谁错了？**
   - 上游定义明确且合理，当前产物偏离 → 当前产物错
   - 上游定义模糊/矛盾/过时，当前产物按合理理解执行 → 上游错
   - 无法判断 → 标记「根因待定」，不擅自修复

3. **禁止症状修复**
   - ❌ 上游定义有问题，只修改下游产物来「对齐」→ 这是症状修复，源头仍在
   - ✅ 找到真正的根因，修改源头，再按变更传播协议同步下游

## 执行方法

| 方法 | 负责方 | 适用检查 |
|------|--------|---------|
| **机械检查** | 自动化工具 | 格式、连续性、存在性、数量匹配、前缀一致性 |
| **语义检查** | AI | 正确性、合理性、满足度、映射语义、设计有效性 |

## 审计轮次规则

- **初始审计**：生成/修改后必须执行
- **修复后重新审计**：必须执行**完整重新审计**，不能因为"只改了编号"就跳过语义检查
- **轮次上限**：每轮审计 + 修复最多 **3 轮**。如果第 3 轮后仍然失败，停止自动修复并请求人工决策。

## 审计报告格式

审计完成后必须输出结构化报告：

```markdown
# 审计报告

## 审计对象
- 产物：[路径]
- 类型：[产物类型]
- 版本：[版本号]

## 检查结果
| 维度 | 层级 | 检查项 | 结果 | 依据/备注 |
|------|------|--------|------|----------|
| 结构 | L1 | 标题层级连续 | 通过/不通过/不适用 | ... |
| 数据 | L2 | 术语与全局契约一致 | 通过/不通过/不适用 | ... |
| ... | ... | ... | ... | ... |

## 根因分析（仅不一致时）
- 不一致位置：
- 根因判断：当前产物错 / 上游错 / 版本不匹配 / 根因待定
- 修复方向：

## 审计结论
- [ ] 通过 — 可进入下游
- [ ] 不通过 — 需修复后重新审计
- [ ] 根因待定 — 需人工决策
```

## 危险信号 —— 停下

- 只查格式不查语义（机械检查替代语义检查）
- 自检代替独立审查（生成方兼任审查方）
- "只改了一个编号，不用重审"
- 第3轮审计后仍失败，继续自动修复而不请求人工决策
- 审计报告只有"通过/不通过"，没有引用依据
- 发现不一致时直接修改下游来"对齐"，不追溯根因

**以上任何一条都意味着：审计正在失效。停下。重新执行。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "只改了一个编号，不用重审" | 任何修改都可能引入新缺陷，必须完整重新审计 |
| "时间紧，先合并再修" | 未审计的产物进入下游 = 把缺陷传播给更多人 |
| "自检过了，没问题" | 自检是必要条件，不是充分条件。审计需要独立视角 |
| "这个维度对文档不适用" | 标注「不适用」并说明理由，不可直接跳过 |
| "第3轮了还没过，我再试一次" | 3轮上限是硬性规则，超限必须请求人工决策 |

## 必需子技能

- 在声称审计完成之前：
  - **必需子技能：** 使用 `evidence-before-claims` —— 在声称通过之前验证审计证据
- 如果审计发现缺陷且根因不明：
  - **必需子技能：** 使用 `systematic-problem-solving` —— 在修复之前调查根因
