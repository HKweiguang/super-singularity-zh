# 完整任务执行（Complete Task Execution）

## 概述

**执行用户要求的一切。不少做任何一件。**

本 skill 防止智能体跳步、合并操作或以代表性示例代替完整任务的常见失败模式。当任务很大时，智能体必须分解它并执行每一块——不是总结，不是抽样，不是为偷懒找借口。

## 核心原则

**违反完整性的字面要求就是违反完整性的精神。**

如果用户要求 10 项，就产出 10 项。如果用户要求处理所有边界情况，就处理所有边界情况。如果任务对上下文来说太大，请用户拆分——不要默默截断。

## 执行规则

1. **先分解**：在执行之前，列出请求所隐含的所有子任务或步骤。
2. **全部执行**：完成清单上的每一项。不跳过，不合并，不用「类似地」来走捷径。
3. **标记完成**：每个子任务完成后，明确确认它已完成。
4. **报告缺口**：如果某个子任务因技术原因无法完成，明确说明阻塞原因并询问用户如何处理。绝不默默遗漏。
5. **大任务必须分步，禁止减项**：当检查项、子任务、待处理文件或预期输出数量超过 **10** 时，**禁止**单次响应中自行减项或抽样。必须输出书面分步计划，按 `dispatching-parallel-agents` 拆分为子任务，或按顺序步骤逐条执行。

## 危险信号 —— 停下并重新评估

- "其余遵循相同的模式……"
- "为了简洁，我只展示一个例子……"
- "类似地，你可以把这个应用到……"
- "这是其他的代表……"
- 因为某个步骤看起来「显而易见」或「重复」就跳过
- 因为「用户可能不需要这个细节」就缩小范围
- 把多个不同的操作合并成一个

**以上任何一条都意味着：你即将违反完整性。停下。执行完整任务。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "这很重复" | 重复是任务的一部分。执行它。 |
| "用户不会在意这个细节" | 你不是用户。不要替他们决定。 |
| "上下文不够了" | 请用户拆分任务。不要默默截断。 |
| "一个例子就够了" | 除非用户明确要求只给一个例子。 |
| "精神比字面重要" | 违反字面就是违反精神。 |
| "我记下来然后继续" | 记下来 ≠ 执行了。去做。 |

## 必需子技能

在声称整个任务完成之前：
- **必需子技能：** 使用 `evidence-before-claims` —— 用新鲜证据验证

如果执行中发现了不清楚的问题或意外行为：
- **必需子技能：** 使用 `systematic-problem-solving` —— 在修补之前先理解根因

## 当上下文不足时

如果完整任务超出了可用上下文：

1. **不要**默默跳过中间步骤。
2. **不要**用总结代替完整执行。
3. **要**暂停执行并询问用户：「这个任务有 N 个步骤。我在当前上下文窗口中可以完成步骤 X-Y。我应该继续处理这些步骤，还是你希望以其他方式拆分任务？」

## 快速参考

| 情况 | 行动 |
|------|------|
| 要求 10 个类似项 | 产出全部 10 个。不走捷径。 |
| 指定了边界情况 | 显式处理每一个边界情况。 |
| 大任务，上下文不足 | 请用户拆分。不要截断。 |
| 用户说「等等」或「诸如此类" | 澄清范围。不要假设你做完了。 |
| 步骤看起来显而易见 | 无论如何执行它。你觉得显而易见 ≠ 做完了。 |
