# 产物审查（Artifact Review）

## 概述

**未经审查的产物不是完成，是草稿。**

本 skill 是产物审查的**统一入口**。每次审查只针对**一个产物目标**——不混审、不跳步、不缩减维度。

产物完成 ≠ 产物可用。审查是交付的必要条件。

**核心原则：** 双向审查。生成方主动请求审查，不自满；审查方技术严谨，不敷衍。

## 铁律

```
NO ARTIFACT ACCEPTED WITHOUT REVIEW
```

如果产物没有经过审查，它就不能被接受、不能进入下游、不能声称完成。停下。请求审查。

## 前置条件

- 产物必须通过 `evidence-before-claims` 验证，确认输出符合预期
- 未经验证的产物不能进入本审查流程

## 审查约束

### 一次只审查一个产物

- **单一目标**：每次审查**只出一份报告**，主角是一个产物；但审查过程中可以查阅上游产物作为参考
- **不混审**：不要把多个产物混在同一个审查中
- **独立判定**：每个产物有独立的审查报告和通过/不通过结论

## 审查流程

```
产物生成完成
    ↓
evidence-before-claims（自检验证）
    ↓
验证通过 → 进入审查
    ↓
[准备阶段] 确认产物类型、加载对应模板、提取验收标准
    ↓
[检查清单阶段] 执行 6 维度客观检查
    ↓
[独立审查阶段] 审查者按 5 维度审查
    ↓
输出审查报告（每条：通过 / 不通过 / 不适用 + 依据）
    ↓
生成方处理反馈（接受 / 澄清 / 争议）
    ↓
修改后重新提交（如有修改）
    ↓
审查方确认全部通过
    ↓
产物进入下游 / 归档 / 发布
```

## 审查阶段详解

### 阶段 1：准备

1. 确认产物类型（PRD / 设计 / 技术方案 / 测试计划 等）
2. 加载对应 `type-step.md` 模板
3. 提取验收标准（来自 acceptance-criteria-first）
4. 确认上游依赖状态
   - 读取 `workflow.yaml` 确认当前产物的 `inputs`
   - 检查每个直接上游产物是否存在、是否已通过审查
   - 上游未审查 → 暂停当前审查，提示先完成上游

### 阶段 2：检查清单（生成方或 AI 执行）

执行 6 维度 × 3 层级客观检查：

**6 维度：**
- **结构**：格式、组织、完整性
- **数据**：准确性、一致性、有效性
- **状态**：当前状态、生命周期阶段
- **错误**：处理、覆盖、边界情况
- **性能**：效率、可扩展性
- **安全**：访问、暴露、漏洞

**3 层级：**
- **L1 产物内部**：产物自身的结构和内容完整性
- **L2 相邻产物**：与直接上下游产物的一致性
- **L3 端到端链**：从头到尾的完整可追溯性

**检查项示例（文档/设计类产物）**

| 维度 | 客观检查项 | 主观检查项 |
|------|----------------|-----------------|
| **结构** | 标题层级连续、表格对齐、编号无跳号 | 章节完整性、内容边界清晰、必填项已填 |
| **数据** | 术语前缀一致、编号格式统一、链接可达 | 术语定义与全局契约一致、同一概念名称统一、引用上游存在 |
| **状态** | 版本号格式正确、日期有效 | 生命周期阶段合理、版本与里程碑对齐 |
| **错误** | 异常场景数量 ≥ 主场景数量 × 20% | 边界条件覆盖完整、错误处理有恢复路径 |
| **性能** | 响应时间基线有数值、并发指标有单位 | 基线合理可达成、可扩展性策略明确 |
| **安全** | 敏感字段已标注、权限矩阵无空行 | 数据分类合理、威胁模型覆盖主要攻击面 |

> **裁剪原则**：产物类型不涉及某维度时，标注「不适用」并说明理由，不可直接跳过。

**模板符合性检查（产物类型专属）**

当产物由 `artifact-workflow` 工作流生成时，以下检查项必须加入检查清单：

| 维度 | 层级 | 检查项 | 检查方式 |
|------|------|--------|----------|
| 结构 | L1 | 产物章节结构符合对应 `type-step.md` 模板要求 | 客观：比对模板 check list |
| 结构 | L1 | 模板中标记为 required 的必填字段已填 | 客观：字段存在性检查 |
| 结构 | L2 | 产物内容无越界（未写入上游定义或下游职责的内容） | 语义：对照 `artifact-workflow` §4.2 节点边界规则 |
| 数据 | L2 | 产物引用上游的部分与上游输出一致 | 语义：比对上游产物内容 |

**L2 相邻产物检查**：直接读取 `workflow.yaml` 中的 `inputs`，比对上游产物内容。

**L3 端到端链检查**：依赖链复杂时（存在间接上游），调用 DAG 引擎扫描全链：
```bash
python3 <skill-path>/scripts/dag-engine.py --workflow .artifacts/workflow.yaml --impact <节点ID>
```
确认整条链的完整性和一致性。依赖链简单时（仅直接上游），L2 检查已覆盖，无需额外调用。

**不一致时的根因判断**

L2 和 L3 检查发现不一致时，**不能直接假设「当前产物错了」**。不一致有三种可能：

| 情况 | 根因 | 修复方向 |
|------|------|---------|
| 上游定义正确，当前产物理解错了 | 当前产物 | 修改当前产物 |
| 上游定义有问题，当前产物按错误上游执行 | 上游产物 | 触发回改模式，修改上游并传播 |
| 上下游都正确，但版本不匹配（上游已更新，当前产物未同步） | 版本同步 | 按变更传播协议更新当前产物 |

**判断流程：**
1. **先审上游**：确认上游产物是否已通过检查且状态稳定。上游未检查 → 暂停当前审查，先完成上游。
2. **谁错了？** 上游明确合理但当前偏离 → 当前产物错；上游模糊矛盾 → 上游错；无法判断 → 根因待定。
3. **禁止症状修复**：找到真正根因，修改源头，再同步下游。

**执行方法**

| 方法 | 负责方 | 适用检查 |
|------|--------|---------|
| **AI 检查** | AI | 格式、连续性、存在性、正确性、合理性、满足度、映射语义、设计有效性 |

### 阶段 3：独立审查（审查方职责）

审查者必须是**独立于生成方**的角色（另一个 AI 实例、用户、或项目定义的审查者）。

**审查五维度：**

| 维度 | 检查内容 | 通过标准 |
|------|---------|---------|
| **意图对齐** | 产物是否满足原始需求？ | 每条需求都有对应实现，无偏离 |
| **边界检查** | 是否有越界内容（写了下游该写的）？ | 所有内容在职责边界内，越界内容已标注迁移 |
| **一致性** | 术语、编号、状态是否与全局契约一致？ | 术语表一致，编号连续，状态定义统一 |
| **完整性** | 是否遗漏了上游定义的某个功能点？ | 上游所有功能点在本产物中有对应引用或实现 |
| **可验证性** | 验收标准是否可执行、可判断？ | 每条验收标准满足 Given-When-Then，可独立验证 |

**审查方技术严谨要求：**
- "看起来不错"不是审查，是渎职
- 每条判断必须有依据：引用上游文档、验收标准或全局契约
- 区分事实与观点，给出可执行的修改建议

### 阶段 4：输出审查报告

```markdown
# 审查报告

## 审查对象
- 产物：[路径]
- 类型：[产物类型]
- 版本：[版本号]

## 审查结果
| 阶段 | 维度 | 检查项 | 结果 | 依据/备注 |
|------|------|--------|------|----------|
| 检查清单 | 结构 | 标题层级连续 | 通过/不通过/不适用 | ... |
| 独立审查 | 意图对齐 | 需求覆盖完整 | 通过/不通过/不适用 | ... |
| ... | ... | ... | ... | ... |

## 根因分析（仅不一致时）
- 不一致位置：
- 根因判断：当前产物错 / 上游错 / 版本不匹配 / 根因待定
- 修复方向：

## 审查结论
- [ ] 通过 — 可进入下游
- [ ] 不通过 — 需修复后重新审查
- [ ] 根因待定 — 需人工决策
```

## 处理反馈

### 生成方职责

1. 每条反馈分类：「接受修改」/「需要澄清」/「有争议」
2. 有争议的反馈不得默默忽略，必须提出替代方案或请求裁决
3. 修改后重新提交，必须回应每条反馈

### 审查方职责

1. 修改后执行**完整重新审查**，不能因为"只改了一个编号"就跳过
2. 每轮审查 + 修复最多 **3 轮**。第 3 轮后仍失败，停止自动修复并请求人工决策

## 危险信号 —— 停下并重新评估

- 一次审查混了多个产物
- 产物写完直接说"完成了"，没有请求审查环节
- 审查者说"没什么问题"但没逐条检查五维度
- 生成方对反馈的回应是"这个不重要"或"用户不会在意"
- 审查报告中只有"通过/不通过"，没有引用依据
- 修改后不再重新审查，直接声称完成
- 审查方是生成方自己（自检 ≠ 审查）

## 常见借口

| 借口 | 真相 |
|------|------|
| "我自检过了，没问题" | 自检是必要条件，不是充分条件。审查需要独立视角。 |
| "时间紧，先合并再修" | 未审查的产物进入下游 = 把缺陷传播给更多人。 |
| "这个产物很小，不用审" | 小产物也有边界和一致性要求。 |
| "改一点小问题不需要重新审" | 任何修改都可能引入新缺陷，修改后必须重新审查。 |
| "用户催得急" | 紧急不是跳过质量的理由。告诉用户：正在审查，预计 X 分钟。 |
| "之前类似的产物没审也过了" | 过去的侥幸 ≠ 现在的标准。 |

## 必需子技能

- 审查前确认验收标准已定义：
  - **必需子技能：** 使用 `acceptance-criteria-first` —— 没有标准则审查无据可依
- 审查中执行客观检查：
  - 按上述 6 维度检查清单逐项确认
- 审查完成后确认修改有效：
  - **必需子技能：** 使用 `evidence-before-claims` —— 修改后验证问题确实已解决
- 审查通过后的收尾阶段：
  - **必需子技能：** 使用 `artifact-finishing` —— 明确产物状态（归档/发布/废弃/回滚）

> 注：本 skill 的「审查」是独立的质量关卡，不能由生成方兼任。审查可以是另一个 AI 实例、用户本人、或项目定义的审查角色。
