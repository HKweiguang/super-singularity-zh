# 产物分支隔离（Artifact Isolation）

## 概述

**并行不是混乱的借口。**

本 skill 防止智能体在并行生成或修改多个产物时，让不同特性/版本的修改互相渗透，导致基线漂移、版本混乱、合并冲突无法追溯的失败模式。

**核心原则：** 不同特性/版本的产物必须在独立空间工作。合并前必须有明确的合并计划和冲突审查。

## 铁律

```
NO OVERLAP WITHOUT EXPLICIT MERGE PLAN
```

如果两个产物分支的修改可能互相影响，但没有明确的合并计划，就不能让它们并行推进。停下。建立隔离。

## 执行规则

1. **修改前确认基线版本**
   - 提问："这是基于哪个基线版本？"
   - 记录基线版本号、文件路径、最后稳定状态
   - 如果基线不存在或不清晰 → 先归档当前状态，再开始修改

2. **创建独立工作副本**

   | 场景 | 命名规则 | 存储位置 |
   |------|---------|---------|
   | 新特性开发 | `{产物名}-{特性名}-draft.md` | 与基线同目录或 `.artifacts/drafts/` |
   | 版本升级 | `{产物名}-v{新}.md` | 与基线同目录 |
   | 实验性探索 | `{产物名}-exp-{日期}.md` | `.artifacts/experiments/` |

3. **标注版本关系**
   - 在副本头部明确标注：「基线版本：vX.Y | 分支原因：[特性名/升级说明]」
   - 在基线文件头部标注：「活跃分支：[分支文件名] | 分支原因：[说明]」

4. **独立演进，定期同步**
   - 分支副本独立修改，不影响基线
   - 如果基线有更新（bugfix、补充），评估是否需要同步到分支
   - 同步时必须记录：同步日期、同步内容、同步原因

5. **合并前强制审查**
   - 合并前检查两个分支是否冲突（同一章节、同一编号、同一术语的不同定义）
   - 冲突必须逐条解决，不能默默覆盖
   - 合并后废弃分支副本，更新基线为合并后版本

## 隔离空间规范

```
project-root/
├── .artifacts/
│   ├── drafts/                ← 特性开发草稿
│   │   ├── prd-v2-payment-draft.md
│   │   └── prd-v2-auth-draft.md
│   ├── experiments/           ← 实验性探索
│   │   └── interaction-exp-2026-06-08.md
│   └── archive/               ← 废弃版本归档
│       └── prd-v1-[已废弃].md
└── docs/
    ├── prd-v1.md              ← 基线版本
    └── prd-v2.md              ← 合并后的新版本
```

## 冲突检测清单

合并前逐条检查：

| 检查项 | 冲突示例 | 处理方式 |
|--------|---------|---------|
| **编号冲突** | 分支 A 新增 REQ-005，分支 B 也新增 REQ-005 | 重新编号，确保全局唯一 |
| **术语冲突** | 分支 A 定义"用户"为消费者，分支 B 定义为商户 | 统一术语，或添加限定词 |
| **章节冲突** | 两个分支同时修改 §3 功能需求 | 逐节比对，合并差异 |
| **状态冲突** | 分支 A 标记功能为「已完成」，分支 B 标记为「待设计」 | 按最新事实统一，标注变更历史 |
| **依赖冲突** | 分支 A 的上游是 prd-v1.md，分支 B 的上游是 prd-v2.md | 统一上游基线，或明确分支各自的依赖链 |

## 危险信号 —— 停下并重新评估

- 直接修改基线版本，没有创建分支副本
- 多个特性共用同一个产物文件，修改互相覆盖
- 分支副本没有标注基线版本，无法追溯来源
- 合并时只取最新修改，不检查冲突
- "这个改动很小，不需要隔离"→ 小改动的叠加就是大混乱
- 废弃版本不归档，留在主目录中与活跃版本混淆

**以上任何一条都意味着：隔离机制失效，产物正在互相污染。停下。建立隔离。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "改动很小，不需要隔离" | 小改动 × 多次并行 = 大混乱。隔离是纪律，不是选项。 |
| "副本太多会找不到" | 命名规范和目录结构就是用来解决这个问题的。 |
| "合并太麻烦了" | 合并麻烦是因为前期没有隔离，不是因为隔离本身。 |
| "我先改着，后面再整理" | 事后整理的成本是事前隔离的 10 倍。 |
| "两个分支不冲突" | 不检查就断言不冲突，是赌博。检查完再判断。 |
| "用户急着要" | 紧急不是跳过隔离的理由。快速创建分支，快速合并。 |

## 快速参考

| 情况 | 行动 |
|------|------|
| 要为已有产物添加新特性 | 创建分支副本 `{产物}-{特性}-draft.md`，标注基线版本 |
| 要升级产物大版本 | 创建 `{产物}-v{新}.md`，保留 `v{旧}` 只读，标注 `v{旧} → v{新}` |
| 基线有 bugfix | 评估是否同步到活跃分支，同步后记录日志 |
| 分支准备合并 | 按冲突检测清单逐条检查 → 解决冲突 → 更新基线 → 废弃分支 |
| 发现两个分支严重冲突 | 暂停合并，启动 `systematic-problem-solving` 分析根因 |
| 特性取消 | 将分支副本移至 `.artifacts/archive/`，标注 `[已取消]` 及原因 |
| 实验成功 | 将实验文件升级为正式分支，按合并流程进入基线 |
| 实验失败 | 归档实验文件，标注失败原因，防止未来重复探索 |

## 必需子技能

在创建分支或合并前评估影响：
- **必需子技能：** 使用 `artifact-workflow` —— 确认上下游依赖关系，评估隔离/合并对依赖链的影响

在发现合并冲突时：
- **必需子技能：** 使用 `systematic-problem-solving` —— 分析冲突根因，避免治标不治本的合并

在合并完成后：
- **必需子技能：** 使用 `artifact-finishing` —— 对废弃分支做明确收尾决策（归档/废弃）

> 注：本 skill 的「隔离」是空间隔离 + 流程隔离。空间隔离防止文件级污染，流程隔离防止决策级污染（不确认基线就开始改）。
