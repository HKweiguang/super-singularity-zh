# 产物审查（Review Before Acceptance）

## 概述

**未经审查的产物不是完成，是草稿。**

本 skill 防止智能体在产物生成后自认"一次过"、拒绝审查反馈、或敷衍处理修改意见的失败模式。产物完成 ≠ 产物可用，审查是交付的必要条件。

**核心原则：** 双向审查。生成方主动请求审查，不自满；审查方技术严谨，不敷衍。

## 铁律

```
NO ARTIFACT ACCEPTED WITHOUT REVIEW
```

如果产物没有经过审查，它就不能被接受、不能进入下游、不能声称完成。停下。请求审查。

## 执行规则

### 生成方职责

1. **产物完成后主动请求审查**
   - 不要等待用户说"检查一下"
   - 主动声明："产物已完成，申请审查"
   - 提供审查上下文：产物类型、上游输入、验收标准、已自检结果

2. **提供审查清单**
   - 列出本产物最可能出问题的 3-5 个检查点
   - 引导审查者聚焦高风险区域

3. **处理反馈时不防御**
   - 审查反馈 ≠ 对你个人的否定
   - 每条反馈分类：「接受修改」/「需要澄清」/「有争议」
   - 有争议的反馈不得默默忽略，必须提出替代方案或请求裁决

4. **修改后重新提交**
   - 修改内容必须回应每条反馈
   - 重新审查直到通过

### 审查方职责

1. **技术严谨，拒绝敷衍**
   - "看起来不错"不是审查，是渎职
   - 每条判断必须有依据：引用上游文档、验收标准或全局契约
   - 发现越界内容时明确标注："此处属于下游职责，建议迁移"

2. **区分事实与观点**
   - 事实："§3 功能需求表格中 REQ-005 未在 §5 测试用例中引用"
   - 观点："我觉得这个设计不够优雅"→ 转化为标准："与全局契约 §2 编号规则不一致"

3. **给出可执行的修改建议**
   - 不只说"不对"，要说"怎么改"
   - 建议引用上游文档的哪个章节、哪个编号

## 审查维度

| 维度 | 检查内容 | 通过标准 |
|------|---------|---------|
| **意图对齐** | 产物是否满足原始需求？ | 每条需求都有对应实现，无偏离 |
| **边界检查** | 是否有越界内容（写了下游该写的）？ | 所有内容在职责边界内，越界内容已标注迁移 |
| **一致性** | 术语、编号、状态是否与全局契约一致？ | 术语表一致，编号连续，状态定义统一 |
| **完整性** | 是否遗漏了上游定义的某个功能点？ | 上游所有功能点在本产物中有对应引用或实现 |
| **可验证性** | 验收标准是否可执行、可判断？ | 每条验收标准满足 Given-When-Then，可独立验证 |

## 审查流程

```
产物生成完成
    ↓
生成方自检（按审查维度逐条检查）
    ↓
提交审查申请 + 上下文 + 自检结果
    ↓
审查方按五维度审查
    ↓
输出审查报告（每条：通过 / 不通过 / 不适用 + 依据）
    ↓
生成方处理反馈（接受 / 澄清 / 争议）
    ↓
修改后重新提交（如有修改）
    ↓
审查方确认全部通过
    ↓
产物进入下游 / 归档 / 发布
```

## 危险信号 —— 停下并重新评估

- 产物写完直接说"完成了"，没有请求审查环节
- 审查者说"没什么问题"但没逐条检查五维度
- 生成方对反馈的回应是"这个不重要"或"用户不会在意"
- 审查报告中只有"通过/不通过"，没有引用依据
- 修改后不再重新审查，直接声称完成
- 审查方是生成方自己（自检 ≠ 审查）

**以上任何一条都意味着：审查流程失效。停下。重新执行。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "我自检过了，没问题" | 自检是必要条件，不是充分条件。审查需要独立视角。 |
| "时间紧，先合并再修" | 未审查的产物进入下游 = 把缺陷传播给更多人。 |
| "审查者不懂业务" | 审查者的职责是检查一致性、完整性、可验证性，不是替代业务判断。 |
| "改一点小问题不需要重新审" | 任何修改都可能引入新缺陷，修改后必须重新审查。 |
| "用户催得急" | 紧急不是跳过质量的理由。告诉用户：正在审查，预计 X 分钟。 |
| "之前类似的产物没审也过了" | 过去的侥幸 ≠ 现在的标准。 |

## 快速参考

| 情况 | 行动 |
|------|------|
| 产物刚写完 | 自检 → 提交审查申请 |
| 审查者发现越界内容 | 标注「建议迁移至下游产物 X」，不在当前产物中默默兼容 |
| 审查意见有争议 | 提出替代方案或请求第三方裁决，不得默默忽略 |
| 修改后重新提交 | 明确列出每条反馈的处理方式，便于审查者快速确认 |
| 用户说"不用审了，直接发" | 解释审查是协议要求，提供审查预计时间 |
| 审查通过后发现遗漏 | 启动回改模式：暂停下游 → 修正上游 → 级联更新 |

## 必需子技能

在审查前确认产物已完成自检：
- **必需子技能：** 使用 `acceptance-criteria-first` —— 审查的输入是验收标准，没有标准则审查无据可依

在审查完成后确认修改有效：
- **必需子技能：** 使用 `evidence-before-claims` —— 修改后验证问题确实已解决

在审查通过后的收尾阶段：
- **必需子技能：** 使用 `artifact-finishing` —— 明确产物状态（归档/发布/废弃/回滚）

> 注：本 skill 的「审查」是独立的质量关卡，不能由生成方兼任。审查可以是另一个 AI 实例、用户本人、或项目定义的审查角色。
