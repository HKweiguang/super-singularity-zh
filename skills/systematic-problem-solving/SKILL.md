# 系统性问题解决（Systematic Problem Solving）

## 概述

随机解决方案浪费时间并制造新问题。快速补丁掩盖了底层问题。

**核心原则：** 在提出解决方案之前，**始终**找到根因。症状修复就是失败。

**违反本流程的字面要求就是违反问题解决的精神。**

## 铁律

```
没有根因调查，不得提出解决方案
```

如果你还没完成第一阶段，就不能提出解决方案。

## 何时使用

用于**任何**原因不明的情况：
- 意外结果
- 失败或错误
- 性能问题
- 预期与实际结果之间的差异
- 反复出现的问题

**在以下情况尤其要使用：**
- 时间压力下（紧急情况让人想猜）
- "就一个快速修复"看起来很明显
- 你已经尝试了多个解决方案
- 之前的方案没起作用
- 你还没完全理解问题

**不要跳过，即使：**
- 问题看起来简单（简单问题也有根因）
- 你很急（匆忙保证返工）
- 有人说现在就修（系统性比乱撞更快）

## 四个阶段

你必须在进入下一阶段之前完成每个阶段。

### 第一阶段：根因调查

**在尝试任何解决方案之前：**

1. **收集证据**
   - 不要跳过异常或警告
   - 完整阅读错误消息、日志或报告
   - 记录具体细节：时间戳、数值、条件
   - 记录问题出现之前发生了什么变化

2. **稳定复现**
   - 你能让问题再次发生吗？
   - 在什么确切条件下？
   - 什么输入或状态会触发它？

### 第二阶段：隔离

1. **缩小范围**
   - 能复现问题的最小场景是什么？
   - 涉及哪些组件？
   - 消除什么之后问题仍然出现？

2. **识别边界**
   - 在什么情况下**不**发生？
   - 正常和异常情况之间有什么不同？

### 第三阶段：假设

**在形成假设之前，必须先经过辩证审视（遵循 `dialectical-thinking` 的四个维度）：**

然后：

1. **形成可检验的假设**
   - 什么可能导致这个证据？
   - 列出**所有**合理的解释，不只是显而易见的（包括与直觉相反的假设）
   - 按可能性和可检验性排序

2. **设计快速测试**
   - 你如何证明或证伪每个假设？
   - 从成本最低的测试开始

### 第四阶段：验证

1. **检验你的主要假设**
   - 运行测试
   - 证据支持还是反驳它？
   - 如果被反驳：返回第三阶段

2. **确认根因**
   - 你能解释**为什么**会发生吗？
   - 修复根因能消除问题吗？
   - 修复在所有相关条件下都能保持吗？

## 危险信号 —— 停下并重新评估

- 在完成第一阶段之前就提出解决方案
- "先试试这个看看"而不理解为什么
- 在不确认根因的情况下修复症状
- 因为「它显然失败了」就跳过复现
- 接受第一个看似合理的解释
- 停在相关关系而不证明因果关系

**以上任何一条都意味着：你在猜测，不是在解决。停下。先调查。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "我知道怎么修" | 如果你知道，你就有证据。检验你的假设。 |
| "我先试试这个" | 不理解就尝试是赌博。 |
| "显然是 X 引起的" | 显而易见常常是错的。验证。 |
| "我们没时间调查" | 乱撞比系统性分析更花时间。 |
| "之前的修复有效" | 之前的修复可能只是掩盖了症状。 |
| "这是个简单问题" | 简单问题也有根因。 |
| "我修完再调查" | 修完之后压力就消失了。调查永远不会发生。 |
| "别人可以去调查" | 如果你在解决它，你就拥有根因。 |

## 快速参考

| 情况 | 行动 |
|------|------|
| 意外结果 | 第一阶段：收集证据 → 第二阶段：隔离 |
| 反复出现的问题 | 第一阶段：检查之前的修复处理的是症状还是原因 |
| 时间压力下 | 第一阶段到第四阶段可以压缩，但**绝不**跳过 |
| 多个假设 | 先测试最便宜的/最简单的 |
| 修复有效但不知道为什么 | 你**还没做完**。理解为什么是必需的。 |

## 必需子技能

在确定根因并应用修复之后：
- **必需子技能：** 使用 `evidence-before-claims` —— 在声称成功之前验证问题确实已解决
