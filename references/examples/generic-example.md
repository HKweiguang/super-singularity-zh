## §1 项目概述

在售后工单量年增 200%、客户投诉响应超时率达 15% 的背景下，智能工单系统是面向制造业中型企业的售后协作平台，核心解决"客户报修后内部流转不透明、响应超时无人负责"的痛点。成功指标：工单平均处理时效从 72h 降至 24h，客户满意度从 82% 提升至 90%。

---

## §2 产物类型定义

| 产物类型 | 职责定位 | 上游依赖 | 下游产物 | 边界红线 |
|---------|---------|---------|---------|---------|
| PRD | 定义产品功能需求、用户故事、业务规则 | 项目顶层定义 | 交互设计、技术方案、测试计划 | 不写页面布局、不写接口定义、不写测试用例 |
| 交互设计 | 定义页面结构、状态机、页面流转 | PRD | UI 设计、技术方案 | 不写视觉样式、不写技术实现 |
| UI 设计 | 定义视觉规范、组件样式、HTML 原型 | 交互设计 | 技术方案、测试计划 | 不写业务逻辑、不写数据模型 |
| 技术方案 | 定义架构、接口、数据模型、部署方案 | PRD、交互设计、UI 设计 | 代码实现、测试计划 | 不写具体业务规则（已在 PRD 中定义） |
| 测试计划 | 定义测试策略、测试用例、验收标准 | PRD、技术方案、UI 设计 | — | 不写需求定义、不写技术实现 |

## §3 产物组织方式

```
docs/
├── top-level.md              ← 项目顶层定义
├── prd.md                    ← 产品需求文档
├── interaction.md            ← 交互设计
├── ui.md                     ← UI 设计
├── tech.md                   ← 技术方案
└── test.md                   ← 测试计划
```

| 规则 | 说明 |
|------|------|
| 文件名 | 小写，用连字符分隔，不含空格 |
| 版本控制 | 产物文件纳入 Git，产物历史通过 Git 追溯 |
| 产物头部 | 每份产物文件头部必须包含：版本号、制定日期、作者、上游文档表 |

## §4 全局术语表

| 术语 | 定义 | 使用场景 | 禁止别名 |
|------|------|---------|---------|
| 工单 | 客户提交的服务请求单，包含问题描述、优先级、处理状态 | PRD、交互设计、技术方案 | 不可称"任务单""服务单" |
| 客户分级 | 根据消费金额和合作时长将客户分为 A/B/C 三级 | 客户管理模块、销售策略 | 不可称"VIP等级""用户等级" |
| 响应时效 | 从客户提交工单到首次人工响应的时间间隔 | PRD、业务规则、验收标准 | 不可称"处理时效""回复时效" |
| 工单池 | 未分配处理人的工单集合，按优先级和技能标签排序 | PRD、技术方案 | 不可称"待办列表""工单队列" |

## §5 全局状态值

| 字段名 | 状态值 | 说明 | 使用场景 |
|--------|--------|------|---------|
| 工单状态 | PENDING | 待分配 | 刚创建，尚未分配给处理人 |
| 工单状态 | PROCESSING | 处理中 | 已分配，处理人正在跟进 |
| 工单状态 | RESOLVED | 已解决 | 问题已处理，等待客户确认 |
| 工单状态 | CLOSED | 已关闭 | 客户确认或超时自动关闭 |
| 工单状态 | ESCALATED | 已升级 | 处理人无法解决，升级至高级技术支持 |
| 优先级 | URGENT | 紧急 | 影响生产或造成重大损失 |
| 优先级 | HIGH | 高 | 影响业务正常运行 |
| 优先级 | NORMAL | 普通 | 一般问题 |
| 优先级 | LOW | 低 | 咨询类、建议类 |

## §6 编号引用契约

| 编号类型 | 前缀 | 格式 | 分段规则 | 连续性要求 | 跳号/作废策略 | 下游产物 | 下游位置 | 是否强制 |
|---------|------|------|---------|-----------|--------------|---------|---------|---------|
| 功能编号 | — | `{模块}-{业务端}-{序号}`<br>如 `TICKET-CONSUMER-001` | 按模块+业务端分段 | 模块+业务端内连续 | 删除留空，不回收 | 技术方案 §4 | 接口表格「对应功能点」列 | 强制 |
| 故事编号 | STORY | `STORY-{模块}-{业务端}-{序号}` | 按模块+业务端分段 | 模块+业务端内连续 | 删除留空，不回收 | 技术方案 §4 | 接口表格「关联故事」列 | 强制 |
| 规则编号 | RULE | `RULE-{模块}-{业务端}-{序号}` | 按模块+业务端分段 | 模块+业务端内连续 | 删除留空，不回收 | 技术方案 §6 | 核心流程步骤引用 | 强制 |
| 错误码 | ERR | `ERR-{模块}-{业务端}-{序号}` | 按模块+业务端分段 | 模块+业务端内连续 | 删除留空，不回收 | 技术方案 §7 | 异常处理表格 | 强制 |
| 全局错误码 | ERR-GLOBAL | `ERR-GLOBAL-{序号}` | 全局统一分段 | 全局内连续 | 删除留空，不回收 | 模块 PRD §7 | 全局错误码引用表格 | 按需引用 |
| 验收项编号 | ACC | `ACC-{模块}-{业务端}-{序号}` | 按模块+业务端分段 | 模块+业务端内连续 | 删除留空，不回收 | 技术方案 §4 | 接口表格「覆盖验收项」列 | 强制 |
| 验收项编号 | | | | | | 测试计划 §1 | 用例表格「关联验收项」列 | 强制 |

> **模块缩写**：TICKET（工单）、CUSTOMER（客户）、REPORT（报表）
> **业务端标识**：CONSUMER（消费者端）、OPERATOR（运营端）、ADMIN（管理端）

## §8 明确不做清单

- 不支持多币种结算 — 理由：当前用户 100% 为国内客户，外币需求仅 2 例，v2 再评估
- 不做移动端 App — 理由：首期以 H5 验证模式，App 开发周期 3 个月，与 MVP 目标冲突
- 不做实时音视频客服 — 理由：当前主要通过文字和电话沟通，音视频需求占比 < 1%，ROI 过低
