# 软件工程 —— PRD 产物步骤模板

> 定位：`generic-step.md` 在**软件工程领域**的具体化示例。展示如何将通用产物框架应用于 PRD（产品需求文档）的生成。
>
> 适用范围：软件工程项目的 PRD 产物。复制到 `.artifacts/steps/prd-step.md`，根据项目需求调整。
>
> **注意**：本模板是通用框架的示例，非完整规范。如需工业级 PRD 模板，参见 `e2e-solution-guard/references/steps/prd-step.md`。

---

## §1 产物头部标准

```markdown
# PRD-v{主版本号}-{模块名}

**版本号**：v{主版本号}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
| 文档 | 类型 | 引用范围 |
|------|------|---------|
| 项目顶层定义 | 规范继承 | 全局规范（术语、状态值、编码规则、产物类型定义等） |
```

---

## §2 章节结构

| 章节 | 说明 | 必填 |
|------|------|------|
| §1 背景与目标 | 要解决的问题、产品目标、成功指标 | 是 |
| §2 用户与场景 | 目标用户、用户故事、使用场景 | 是 |
| §3 功能需求 | 以用户故事为索引的功能需求 | 是 |
| §4 非功能需求 | 性能、安全、可扩展、合规、可用性 | 是 |
| §5 数据模型 | 实体、字段、类型、约束 | 否 |
| §6 业务规则 | 状态流转、权限、计算逻辑、校验规则 | 是 |
| §7 错误处理 | 错误码、触发场景、前端提示 | 否 |
| §8 验收标准 | 可测试的条件+操作+预期结果 | 是 |
| §9 依赖与范围 | 上游依赖、技术约束、业务假设、明确不做 | 是 |
| §10 附件 | 参考资料、原型链接 | 否 |

> **章节稳定性**：以下章节编号必须固定，以确保下游产物（交互设计、技术方案、测试）的引用稳定性。如无内容，保留章节标题并声明"跳过"，**不可直接删除章节导致编号变化**。

---

## §3 产物边界

**PRD 只定义"做什么"和"为什么做"**。以下内容的职责归属其他产物：

- **交互设计**（页面结构、状态机、页面流转）→ `interaction-step.md`
- **视觉设计**（色彩、字体、间距、组件规范）→ `ui-step.md`
- **技术实现**（架构、接口、数据模型细节）→ `tech-step.md`
- **测试策略**（测试用例、测试范围）→ `test-step.md`

**越界检测**：若 PRD 中出现交互描述（"点击XX按钮后跳转到XX页面"）、视觉描述（"使用XX颜色"）、技术描述（"调用XX接口"），或独立的「页面概览/页面清单」章节，应标记为**越界内容**，提示迁移。

## §4 各章节格式要求

### 4.1 背景与目标

**内容**：回答三个问题——**要解决什么问题**、**达成什么目标**、**如何衡量成功**。

**格式建议**：3-5 段自由文本，必须包含成功指标。

**成功指标**（必填）：
- 至少定义 1 个可量化的成功指标
- 指标应覆盖业务价值（用户侧）和系统价值（运营侧）

---

### 4.2 用户与场景

**内容**：定义模块的目标用户、核心用户故事、关键使用场景。

**格式要求**：
- **目标用户**：1-2 段描述，包含用户角色、职责、痛点
- **用户故事**：**必须用表格**。列：故事编号 | 用户故事 | 优先级 | 对应功能编号

**用户故事格式**：
```
As a [用户角色], I want [目标], so that [收益]
```

---

### 4.3 功能需求

**内容**：以用户故事为索引，逐条定义功能需求。每个功能必须关联用户故事、业务规则和验收标准。

**格式要求**：**必须用表格**。列：功能编号 | 故事编号 | 功能名称 | 优先级 | 交互方式 | 业务规则 | 验收标准。

**字段规范**：
- 功能编号：格式继承项目顶层定义 §6 编号引用契约
- 功能名称：动宾结构，建议不超过 10 字
- 优先级：P0（必须）/ P1（重要）/ P2（加分）
- 交互方式：功能以什么形态呈现（页面、弹窗、API、后台任务），不写具体布局
- 业务规则：填写对应 §6 的规则编号
- 验收标准：填写对应 §8 的验收项编号

---

### 4.4 非功能需求

**内容**：定义功能需求之外的系统属性要求。

**格式建议**：表格。列：类型 | 需求描述 | 衡量标准 | 优先级。

---

### 4.5 数据模型

**内容**：模块涉及的所有业务实体及其字段定义。

**格式**：每个实体一张表，列：字段 | 业务类型 | 约束 | 业务说明。

**边界**：不写数据库实际类型、索引策略、分表策略、SQL。

---

### 4.6 业务规则

**内容**：状态流转、权限规则、计算逻辑、校验规则。

**格式**：表格（规则编号 | 规则描述 | 触发条件 | 处理逻辑）或状态机图。

**边界**：不写代码实现、不写前端交互。

---

### 4.7 错误处理

**内容**：模块内错误码的定义，以及从顶层定义继承的全局错误码引用。

**格式**：
- **模块特有错误码**：表格，列：错误码 | 触发场景 | 前端提示
- **全局错误码引用**：表格，列：错误码 | 来源 | 触发场景 | 前端提示（可覆盖）

**边界**：不写 HTTP 状态码、后端日志级别、异常类名。

---

### 4.8 验收标准

**内容**：每个功能的可测试标准。

**格式要求**：**必须用表格**。列：验收项编号 | 功能编号 | 验收项 | 条件 | 操作 | 预期结果。

每条验收项必须用 `[条件] + [操作] + [预期结果]` 句式描述。

**边界**：不写测试步骤、不写测试数据、不写性能指标。

---

### 4.9 依赖与范围

**内容**：声明模块的外部依赖、技术约束、业务假设，以及当前版本明确不做的范围。

**格式建议**：4 个子节，每个子节用列表或表格。

**子节**：
- **上游依赖**：本模块依赖的其他模块/服务/数据源
- **技术约束**：已知的技术限制
- **业务假设**：产品决策基于的假设
- **明确不做**：本版本明确排除的功能/场景，每项附理由

---

### 4.10 附件

**内容**：参考资料、原型链接、相关文档链接。

**格式**：自由文本，只放链接和简要说明。

---

## §5 格式规范

- 编号继承项目顶层定义 §6 编号引用契约
- 状态值继承项目顶层定义 §5 全局状态值
- 术语继承项目顶层定义 §4 全局术语表
- 表格使用标准 Markdown 语法，禁止合并单元格

## §6 检查清单

> 以下仅列出高优先级 `[模型]` 检查项（需 AI 语义判断），`[脚本]` 项由 audit-engine 自动执行：

- [ ] **L1** `[模型]` §6 权限规则引用顶层定义 §2 权限级别定义，未将权限级别拆分为独立业务端
- [ ] **L1** `[模型]` 字段名统一使用业务术语，不混用技术术语
- [ ] **L1** `[模型]` §3/§5/§6 中首次出现的业务术语标注来源
- [ ] **L1** `[模型]` §6 状态流转有明确的触发条件和目标状态
- [ ] **L1** `[模型]` §7 错误码触发场景具体可执行（非"用户操作时"）
- [ ] **L1** `[模型]` §4 性能指标可量化（有数字和阈值）
- [ ] **L1** `[模型]` §4 安全需求具体可执行（非"保证数据安全"）
- [ ] **L1** `[模型]` 产物中无不属于 PRD 范畴的交互/视觉/技术描述
