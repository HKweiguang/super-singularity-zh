# 通用产物步骤模板

> 定位：定义某类产物的生成规范，包括头部格式、章节结构、边界规则和审计要求。
>
> 适用范围：任何产物类型。复制到 `.artifacts/steps/{type}-step.md`，根据产物类型和项目需求修改。
>
> 使用方式：产物生成/修改时，`artifact-workflow` 自动加载对应步骤模板作为骨架。

---

## §1 产物头部标准

每份产物文件头部必须包含：

```markdown
# {产物名称}

**版本号**：v{主版本}.{补丁号}
**制定日期**：YYYY-MM-DD
**作者**：姓名

**上游文档**：
| 文档 | 类型 | 引用范围 |
|------|------|---------|
| {上游文档路径/名称} | 规范继承 / 需求输入 / 设计输入 | {具体章节/编号/范围} |
```

- `类型` 必须明确是**规范继承**还是**输入依赖**
- `引用范围` 必须具体到章节、编号或功能点，不能写"全文"
- 修改时如上游有更新，必须同步更新此表

---

## §2 章节结构

| 章节 | 说明 | 必填 |
|------|------|------|
| §1 概述 | 背景、目标、范围 | 是 |
| §2 输入与约束 | 上游文档、全局契约、技术约束 | 是 |
| §3 核心内容 | 根据产物类型定义的具体内容 | 是 |
| §4 边界与例外 | 明确不做、例外场景、限制条件 | 否 |
| §5 验收标准 | 可验证的完成条件 | 是 |
| §6 附件 | 参考资料、链接 | 否 |

> **章节稳定性**：章节编号固定，确保下游产物引用稳定性。如无内容，保留标题并声明"跳过"或"当前版本无此项"，**不可直接删除章节导致编号变化**。

---

## §3 各章节写作要求

### 3.1 概述

**内容**：回答三个问题——**要解决什么问题**、**达成什么目标**、**如何衡量成功**。

**格式建议**：3-5 段自由文本，必须包含成功指标。

**成功指标**（必填）：
- 至少定义 1 个可量化的成功指标
- 指标应覆盖业务价值和执行价值

---

### 3.2 输入与约束

**内容**：声明本产物依赖的上游文档、全局契约引用、技术/业务约束。

**格式建议**：
- **上游文档**：表格，列：文档 | 类型 | 引用范围
- **全局契约引用**：列出引用的顶层定义中的术语、状态值、编号规则
- **约束条件**：技术限制、业务假设、合规要求

---

### 3.3 核心内容

**内容**：根据产物类型定义的具体内容。这是产物的主体部分。

**格式建议**：由产物类型决定。常见格式包括：
- 表格（功能清单、接口定义、检查项等）
- 列表（步骤、规则、假设等）
- 自由文本（分析、说明、背景等）
- 结构化数据（YAML/JSON 代码块等）

**通用要求**：
- 使用标准 Markdown 语法
- 表格禁止合并单元格
- 编号继承项目顶层定义 §6 编号引用契约

---

### 3.4 边界与例外

**内容**：明确本产物不覆盖的范围、例外场景和限制条件。

**格式建议**：列表。每项格式：`{不做的事项} — 理由：{为什么不做的简要说明}`。

> 无明确不做事项则写：**当前版本无明确不做事项。**

---

### 3.5 验收标准

**内容**：定义本产物的可验证完成条件。每条验收标准必须可被独立验证。

**格式建议**：表格。列：验收项编号 | 验收项 | 条件 | 操作 | 预期结果。

每条验收项必须用 `[条件] + [操作] + [预期结果]` 句式描述。

**示例**：

| 验收项编号 | 验收项 | 条件 | 操作 | 预期结果 |
|-----------|--------|------|------|---------|
| ACC-001 | 登录功能可用 | 用户已注册 | 输入正确的用户名和密码，点击登录 | 系统跳转至首页，显示用户信息 |

---

### 3.6 附件

**内容**：参考资料、原型链接、相关文档链接。

**格式**：自由文本，只放链接和简要说明。

---

## §4 产物边界

**本产物不写**：
- 已在上游定义的内容（不写已在上游产物中明确的事项）
- 属于下游职责的内容（不写应由下游产物处理的事项）
- 超出当前产物类型范围的内容

**发现越界内容时**：标记并提示迁移，不在当前产物中默默兼容。

---

## §5 审计要求

产物生成/修改后必须执行以下审计流程：

```
机械检查（脚本）→ 未通过 → 修复 → 必须重新机械检查
   ↓ 通过
语义检查（AI）→ 未通过 → 修复 → 重新语义检查
   ↓ 通过
审查（review-before-acceptance）→ 四选一决策（artifact-finishing）
```

### 5.1 机械检查

```bash
python3 scripts/audit-engine/cli.py --file <产物路径> --type <产物类型>
```

- **硬性规则**：机械未通过禁止启动语义检查
- **修复后复验**：机械修复后必须对该产物重新执行完整机械检查

### 5.2 语义检查

按 `checklists/{type}-checklist.md` 逐项判断：

- **L1 检查项**：阻塞性，必须全部通过
- **L2 检查项**：重要，无阻塞性问题即可
- **L3 检查项**：建议性，记录但不阻塞

### 5.3 轮次控制

- 单份产物审计+修复最多 **3 轮**（含初次审计）
- 第 3 轮仍不通过 → 停止自动修复，向用户呈现剩余问题，请求人工决策

## §6 变更传播

### 6.1 修改本产物时的上游检查

```
修改本产物
    ↓
读取所有上游文档
    ↓
逐条比对修改内容是否在上游定义范围内
    ├─ 未超出 → 直接修改
    └─ 超出 → 立即停止，先改上游
```

### 6.2 修改本产物时的下游影响

```
修改本产物
    ↓
运行 dag-engine --impact {本产物文件} --dir {产物目录}
    ↓
列出影响范围清单
    ↓
询问用户是否级联修改
    ├─ 用户确认 → 同步修改所有受影响下游
    └─ 用户拒绝 → 在受影响下游标注 [待同步]
```

---

## §7 格式规范

- 编号继承项目顶层定义 §6 编号引用契约
- 术语继承项目顶层定义 §4 全局术语表
- 状态值继承项目顶层定义 §5 全局状态值
- 表格使用标准 Markdown 语法，禁止合并单元格
- 产物文件使用 UTF-8 编码，换行符为 LF

---

## §8 检查清单

> 详细检查清单参见 `references/checklists/generic-checklist.md`。
>
> 以下仅列出本产物类型的高优先级 `[模型]` 检查项（需 AI 语义判断），`[脚本]` 项由 audit-engine 自动执行：

- [ ] **L1** `[模型]` 产物头部包含版本号、制定日期、作者、上游文档表
- [ ] **L1** `[模型]` 上游文档表的"类型"明确区分"规范继承"和"输入依赖"
- [ ] **L1** `[模型]` 上游文档表的"引用范围"具体到章节/编号，不写"全文"
- [ ] **L1** `[模型]` §3 核心内容中使用的术语与顶层定义 §4 一致，无别名混用
- [ ] **L1** `[模型]` §3 核心内容中使用的状态值与顶层定义 §5 一致
- [ ] **L1** `[模型]` 编号格式符合顶层定义 §6 编号引用契约
- [ ] **L1** `[模型]` 产物中无不属于本产物类型范围的内容（越界检测）
- [ ] **L1** `[模型]` §5 验收标准每条均可被独立验证（条件+操作+预期结果）
- [ ] **L2** `[模型]` 检查上游顶层定义在本产物中的完整映射，发现缺失标记为阻塞性差异
- [ ] **L2** `[模型]` 章节编号连续，无跳号，无直接删除导致的编号变化
